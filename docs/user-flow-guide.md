## Overview

Rail is an automated wealth system designed around one core principle: **money starts working the moment it arrives**. This document maps the complete user journey from app download to active wealth building, showing how Rail eliminates financial decision-making through intelligent automation.

### Core System Rule

Every deposit into Rail is automatically split:

- **70% â†’ Spend Balance** (everyday expenses)
- **30% â†’ Invest Balance** (automatic allocation)

This rule is system-defined, always active, and not user-editable in the MVP.

### Target User

**The Indecisive Optimist (18-26)**

- Wants wealth progress without active participation
- Avoids financial decision-making
- Values speed and simplicity over control

---

## Core User Journey

The primary Rail experience follows this minimal path:

```
Download App â†’ Sign Up â†’ Load Money â†’ Money Starts Working
```

Everything beyond this is internal system execution. There are no branching paths, optional setup steps, or strategic decisions required from the user.

### Journey Phases

| Phase             | Duration     | User Action             | System Response              |
| ----------------- | ------------ | ----------------------- | ---------------------------- |
| **Discovery**     | 30 seconds   | Download from App Store | App launches in < 2 seconds  |
| **Onboarding**    | < 2 minutes  | Apple Sign-In + KYC     | Auto-create account & wallet |
| **First Funding** | < 60 seconds | Load money (any method) | Auto-split 70/30 + deploy    |
| **Ongoing Usage** | Continuous   | Spend normally          | Auto-invest round-ups        |

## User Experience Principles

Rail's experience is designed around **trust through action, not education**. Every interaction reinforces the core philosophy: money should start working the moment it arrives.

### Core UX Principles

| Principle                               | Implementation                      | Example                                              |
| --------------------------------------- | ----------------------------------- | ---------------------------------------------------- |
| **Fewer decisions increase confidence** | System-defined 70/30 split          | No allocation sliders or percentage inputs           |
| **Speed creates belief**                | < 60 second deposit processing      | Money moves immediately, no delays                   |
| **Defaults outperform settings**        | Round-ups ON by default             | Simple toggle, no configuration options              |
| **State matters more than detail**      | Show "Active" not individual trades | Station shows system status, not portfolio breakdown |

### Language Guidelines

| Use This           | Not This                    | Reason                             |
| ------------------ | --------------------------- | ---------------------------------- |
| "Load money"       | "Deposit funds"             | More casual, less financial jargon |
| "Money is working" | "Portfolio is performing"   | Focuses on action, not returns     |
| "Your Rail"        | "Your account"              | Brand-specific, more personal      |
| "System is active" | "Investments are allocated" | Emphasizes automation              |

### Screen Design Rules

1. **If a screen requires explanation, it's wrong**
2. **If a user hesitates, something failed**
3. **Show direction, not detail**
4. **Communicate through state changes**

## Error Handling & Edge Cases

### Common Error Scenarios

| Scenario                    | User Experience                          | System Behavior                          |
| --------------------------- | ---------------------------------------- | ---------------------------------------- |
| **KYC Rejection**           | "Verification needed - please try again" | Allow retry with different documents     |
| **Insufficient Funds**      | "Not enough in Spend Balance"            | Block transaction, suggest loading money |
| **Deposit Failure**         | "Deposit processing - we'll notify you"  | Retry webhook processing, manual review  |
| **Trade Execution Failure** | Station shows "Allocating" longer        | Retry trades, fallback to cash position  |
| **Card Declined**           | Standard decline message                 | Check balance, fraud detection           |
| **Network Issues**          | "Connection issue - trying again"        | Offline mode, sync when reconnected      |

## Conclusion

Rail's user flow is intentionally minimal and automated. The system succeeds when users:

1. **Trust the automation** - Money flows in without hesitation
2. **Forget about investing** - Growth happens invisibly
3. **Use Rail as primary spending** - Replace traditional checking account
4. **Don't intervene** - System operates without user management

The entire experience is designed around the principle that **fewer decisions increase confidence**. Every screen, interaction, and system behavior reinforces this philosophy.

If users feel the need to manage, optimize, or control their money within Rail, the product has failed its mission. Success is measured not by engagement with investment features, but by the absence of such engagement - money working automatically while users live their lives.
